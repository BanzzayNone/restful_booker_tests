# Restful Booker API Testing Framework

Автоматизированное тестирование API отеля [restful-booker.herokuapp.com](https://restful-booker.herokuapp.com/apidoc/).

Этот учебный проект демонстрирует мой взгляд и возможности API-тестирования: параметризация, Contract Testing, негативные сценарии и независимость тестов. Ввиду ограничений API пополняться не будет, Playwright и прочие пользуемые мной библиотеки будут представлены в других проектах.

---
# Что внутри:

| Функция              | Описание                                             |
| -------------------- | ---------------------------------------------------- |
| **CRUD-тесты**       | Полное покрытие: Create, Read, Update, Delete        |
| **Параметризация**   | 1 тест → 10+ прогонов с разными данными (Faker)      |
| **Contract Testing** | Валидация структуры ответов через Pydantic           |
| **Негативные тесты** | Валидация ошибок, граничные значения, авторизация    |
| **Независимость**    | Тесты можно запускать в любом порядке и по маркерам  |
| **Генерация данных** | Реалистичные данные через Faker (имена, даты, числа) |
| **Фикстуры**         | Переиспользуемые компоненты: клиент, токен, данные   |

# Установка и запуск

## 1. Виртуальное окружение

``` bash

# Создать виртуальное окружение
python -m venv .venv

# Активировать окружение
# Windows:
.venv\Scripts\activate
# Linux/Mac:
source .venv/bin/activate

# Установить зависимости
pip install -r requirements.txt
```

## 2. Запуск тестов

```bash
# Все тесты
pytest

# Только позитивные тесты
pytest -m "not negative" -v

# Только негативные тесты
pytest -m negative -v

# Только Contract-тесты
pytest -m contract -v

# Только Smoke-тесты (быстрая проверка)
pytest -m smoke -v

# Только тесты с перебором сгенерированных параметров
pytest -m parametrize -v

# С отчётом в HTML
pytest --html=reports/report.html -v
```

## 3. Фильтрация по файлам

```bash
# Конкретный файл
pytest tests/api/test_negative.py -v

# Конкретный тест
pytest -k "test_name_example"

# Конкретный параметр (из параметризации)
pytest "tests/api/test_negative.py::test_create_booking_with_invalid_price[-1]" -v
```

# Основы тест-дизайна в проекте

## 1. Минимум тестов, максимум покрытия

Один параметризованный тест заменяет 10+ отдельных функций. 
При возможности используется сгенерированные через Faker параметры, у таких тестов есть маркер - `parametrize`
## 2. Независимость тестов

- Каждый тест создаёт и удаляет свои данные
- Тесты можно запускать в любом порядке
- Поддержка параллельного запуска (`pytest -n auto`)

## 3. Contract Testing

Валидация структуры ответов через Pydantic-модели для удобства чтения и модификации.

## 4. Централизованная конфигурация

Все настройки в одном месте (`config/settings.py` + `fixtures/`) для быстрого изменения и проверки тестов. Менять параметры за один раз вместо сотни повторений.

# Маркеры

| Маркер        | Описание                                  | Команда запуска         |
| ------------- | ----------------------------------------- | ----------------------- |
| `smoke`       | Быстрая проверка критического функционала | `pytest -m smoke`       |
| `negative`    | Негативные сценарии                       | `pytest -m negative`    |
| `contract`    | Проверка структуры ответов                | `pytest -m contract`    |
| `parametrize` | Проверка тестов с перебором параметров    | `pytest -m parametrize` |

# Касательно ограничений API

## При написании негативных сценариев опытным путём выявил недостатки API. В рамках тестов это повлияло только на ожидаемые статус коды ответа и финальные проверки: 

### 1. Проблема валидации - API не проверяет цену, в негативных сценариях это показано. 
### 2. Проблема с неожиданными сценариями - сервер не имеет прописанной логики на попытки записи пустых значений, в ответ ошибка 500, также видно в негативных сценариях
### 3. Проблема с ответом на неверный/пустой токен авторизации. В негативных сценариях показано.
### 4. Проблема с Content-Type заголовком, нет блокировки при неправильном формате
